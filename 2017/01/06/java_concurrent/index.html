java并发编程中的几种机制

发表于2017-01-06 | 分类于 tech |

多线程环境会遇到什么问题

  • 一个线程修改了变量, 另一个线程看不到该变量的修改(内存可见性)
  • 看起来是单个的操作, 实际执行时可能是多步操作(cpu实际执行的指令), 可能有其它线程在这多步操作未完成时访问相关变量(操作的原子性)
    eg: 对一个64位的32操作
  • (代码块的原子性)

java为并发编程提供了哪几种机制

volatile变量, 使用这些变量时不会发生上下文切换/线程调度
以最轻量级的开销提供了全局可见性
原子变量,
使用场景: 标识状态, 计数器

缺陷: 无法构建复合操作

cas(内存地址, 比较值, 新值)
当内存中的当前值=比较值时, 将新值赋给该内存位置; 如果不一致, 返回失败, cup会针对该操作有专门指令, 原子性通过cpu来保证

可以像锁机制一样, 即通过原子性, 又提供可见性, 且不会有死锁的问题
典型模式:
    先从v中读取a, 根据a计算b, 然后通过cas以原子方式将v中的值由a变成b
    由于cas能检测到来自其他线程的干扰, 因此不使用锁, 也能够实现原子的读-改-写操作序列
  cas的缺点: 需要处理者自己来处理竞争问题(重试or回退or放弃), 而锁会自动处理竞争问题(线程在拿到锁前会一直阻塞)
    使用cas, 可以实现一些不使用锁的非阻塞算法

内部锁: synchronized 关键字, 保证只有持有锁的线程可以执行代码块, 并保证块内操作的内存可见性
锁的底层实现: 每个对象都有一个锁, 也就是监视器(monitor) monitor=0, 表明该对象没有被锁定, monitor=1,
表明有线程持有该对象的锁, monitor>1 表明一个线程多次拿到该对象的锁
线程是持有锁的单位
使用内部锁会有什么问题:
死锁, 并且出现一旦死锁就只能重启了
防止死锁的方法:
    避免有一步操作hold多个锁的情况
    如果非要在一步操作内hold多个锁, 需要保证操作hold锁的顺序, 以保证同不会出现相互等待对方解锁的情况

通过锁界定一个临界区, 同时只允许一个线程进入, 其中的代码最多同时只有1个线程执行,  临界区内容在执行时jvm会加入内存屏障, 以保证临界区内的变量可见性

ThreadLocal : 避免使用共享状态, 效率最高

ReentrantLock : 以独占的方式获取操作
ReentrantLock的问题: 如果忘记在finally块中调用unlock就会导致灾难
什么时候使用ReentrantLock: 需要使用可定时/轮询/中断 获取锁操作的场景, 以及需要使用公平锁的场景


ReadWriteLock:底层实现:cas + 线程notify机制 +
    一个资源可以被多个读操作访问或一个写操作访问, 但两者不能同时进行 读操作更多的场景可以考虑使用ReadWriteLock (比起ReentrantLock, 可以允许同时有多个多操作, 读操作更多时, 可以提升程序的伸缩性)

threadlocal机制

实践:
不要同时使用多种同步机制, 同时使用多种同步机制会带来混乱, 也不会在性能和安全性上带来好处
不要在无法快速完成的操作上加锁

避免密码过长的情况, 应为这样会导致ddos , 单次计算过长的串消耗过多算力
